# Hook Code Signing Policy

This document defines how signed Hook release artifacts are produced, reviewed,
and approved.

It is written to be public, provider-neutral, and compatible with hosted
signing services such as SignPath Foundation or commercial Windows code-signing
providers.

## Scope

This policy applies to signed Windows release artifacts published from:

- repository: `https://github.com/aiaimimi0920/Hook`
- product: **Hook**
- maintainer label: **yamiyu**

The policy is about release integrity, approval, and signing workflow. It does
not replace the end-user package notes in `README.md` or
`UIACCESS_DISTRIBUTION.md`.

## Signed vs unsigned artifacts

Hook may publish more than one Windows package on the same release page:

- **Portable package**
  - intended for quick trial and ordinary daily use
  - may remain unsigned depending on the current release lane
- **Installer / UIAccess-oriented package**
  - intended for the most reliable Windows desktop interaction path
  - must only be published as signed output
  - is designed for installation into a trusted location such as
    `Program Files`

Unsigned artifacts must never be described as equivalent to the signed
UIAccess-oriented installer path.

## Source of truth

Only builds produced from the public GitHub repository are eligible for signing.

The required source-of-truth conditions are:

1. the source commit exists in the public repository history;
2. the release is created from a tagged commit;
3. the signed artifact is generated by the repository workflow, not by a
   private local rebuild;
4. the artifact payload matches the reviewed repository contents for that tag.

## Release tags and build lane

The current release workflow expects version tags in the form:

- `Vx.x.x`

Signed artifacts are built by GitHub Actions from a release tag or an explicit
maintainer-triggered release workflow run that targets that tag.

## Authorized roles

### Committers

Committers are maintainers with write access to the Hook repository who can land
changes on the release branch or create the release tag.

### Reviewers

Reviewers are maintainers who inspect the tagged source, workflow changes, and
release-impacting documentation before a signed release is approved.

In a single-maintainer operating period, the same repository administrator may
act as both committer and reviewer, but the review checklist still applies and
must be completed before approval.

### Approvers

Approvers are maintainers who have access to the configured signing provider and
who are allowed to approve a signing request or release publication.

Approvers must:

- use multi-factor authentication on the relevant GitHub and signing-provider
  accounts;
- verify that the release tag matches the intended public source revision;
- verify that the produced artifact type matches the package being released.

## Approval requirements

Before any signed Hook artifact is approved, the approver must confirm all of
the following:

1. the release version/tag is intentional;
2. the workflow run completed from the expected repository and tag;
3. the artifact came from the hosted build output for that run;
4. no unsigned replacement has been manually swapped into the installer lane;
5. the release notes and package labels clearly distinguish **portable** from
   **installer/UIAccess** output;
6. the relevant public policy documents remain current:
   - `README.md`
   - `UIACCESS_DISTRIBUTION.md`
   - `docs/CODE_SIGNING_POLICY.md`
   - `docs/PRIVACY_POLICY.md`

## Signing environment rules

Signed releases should use a hosted signing service or other managed private-key
storage. Private signing keys must not be committed to this repository.

If GitHub Actions is used for signing:

- signing credentials must be stored as repository or organization secrets;
- signing secrets must only be available to the intended release workflow;
- the signed installer lane must be skipped when signing material is absent,
  instead of publishing an unsigned substitute.

## What gets signed

For the Windows UIAccess-compatible lane, the signed payload is the distributed
`hook.exe` that will be installed by the installer package helper scripts.

If additional signed artifacts are introduced later, this policy must be
updated before they are publicly released as trusted outputs.

## Incident response

If a signing credential, signing account, or release approval path is suspected
to be compromised, Hook maintainers will:

1. stop approving new signed releases;
2. rotate or revoke the affected signing access;
3. remove or clearly mark affected release assets if needed;
4. publish a public maintainer note through GitHub explaining the incident
   status and next safe version.
